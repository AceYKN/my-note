# 数据库规范化流程

## 流程一：考试做题流程 / 项目设计模式

**场景：** 适用于从零开始设计一个真实的数据库系统或处理复杂的案例分析题。其核心在于将现实世界的业务逻辑转化为数据库模型。

### 步骤

**① 问题描述（自然语言）**

将需求方的口头要求或文字需求记录下来。

**② 应用语义归纳**

补充潜在语义（默认），分析业务规则。

> 例如："一个员工只能属于一个部门"，这决定了数据之间的对应关系。

**③ 绘制 E-R 图**

使用实体（框）、属性（圆）和联系（菱形）直观地表示数据结构。

**④ 转换关系模式**

将 E-R 图转化为逻辑表格（关系模式）。

**⑤ 说明范式，说明不合理性**

检查当前的表格属于第几范式（通常是 1NF 或 2NF），并指出存在的数据冗余、插入异常或删除异常。

**⑥ 规范化推导（到 BCNF）**

运用算法将表格拆分，逐步提升范式级别。

**⑦ 给出设计结果**

```
R₁(…)   码：     外码：
R₂(…)   码：     外码：
……
```

**⑧ 结论**

> 经过以上规范化推导，问题已经设计到 BCNF（Boyce-Codd Normal Form），达到函数依赖范畴的最高级范式。

## 流程二：书本上做题流程

**场景：** 适用于纯理论练习题。这类题目通常已经省略了业务分析，直接给出了数学化的定义。

### 步骤

**① 问题描述（自然语言）**

**② 应用语义归纳**

补充潜在语义（默认）。

**③ 形式化定义**

$$R\langle U,\ F\rangle$$

其中：

- $U = \{\dots\}$：属性全集，即表里所有的列
- $F = \{\dots\}$：函数依赖集，即列与列之间的决定关系
- **码：** （Primary Key，能唯一标识一行的最小属性组）
- **非主属性集** $= \{\dots\}$：不包含在任何候选码中的属性

**④ 说明范式，说明不合理性**

**⑤ 规范化推导（到 BCNF）**

**⑥ 规范化结果**

```
R₁(…)   码：     外码：
R₂(…)   码：     外码：
……
```

**⑦ 结论**

> 经过以上规范化推导，问题已经设计到 BCNF，达到函数依赖范畴的最高级范式。
