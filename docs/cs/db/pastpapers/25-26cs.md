## 三、简答题（5 分/题 \* 3 题 = 15 分）

### 1. 题目

将 1NF 的函数关系逐步分解为 BCNF，说明每一步消除了什么类型的函数依赖。

答案与解析：

从 1NF 到 BCNF 的规范化过程如下：

1. 1NF → 2NF

   要求：
   在 1NF 的基础上，每一个非主属性都完全函数依赖于任何一个候选码。

   消除的问题：
   消除非主属性对码的部分函数依赖。

   即若存在：
   $X \subset K$，且 $X \to A$

   其中 $K$ 是候选码，$A$ 是非主属性，则说明 $A$ 部分依赖于码，需要分解。

2. 2NF → 3NF

   要求：
   在 2NF 的基础上，每一个非主属性都不传递依赖于候选码。

   消除的问题：
   消除非主属性对码的传递函数依赖。

   即若存在：
   $K \to X$，$X \to A$，且 $X$ 不是候选码，$A$ 是非主属性

   则有：
   $K \to A$

   这种依赖属于传递依赖，需要分解。

3. 3NF → BCNF

   要求：
   对于关系模式中的每一个非平凡函数依赖 $X \to Y$，$X$ 必须包含候选码，即 $X$ 必须是超码。

   消除的问题：
   消除所有决定因素不是候选码的函数依赖。

   换句话说，BCNF 比 3NF 更严格，不仅要求非主属性不能部分依赖、传递依赖于码，还要求任何函数依赖的决定因素都必须是码。

### 2. 题目

解释共享锁和排他锁的含义，并给出其相容矩阵。

答案与解析：

1. 共享锁，简称 S 锁

   若事务 T 对数据对象 A 加上 S 锁，则事务 T 可以读取 A，但不能修改 A。

   其他事务可以再对 A 加 S 锁，但不能对 A 加 X 锁。

2. 排他锁，简称 X 锁

   若事务 T 对数据对象 A 加上 X 锁，则事务 T 可以读取和修改 A。

   其他事务不能再对 A 加 S 锁，也不能对 A 加 X 锁。

3. 相容矩阵

| 已加锁 \ 申请锁 | S 锁   | X 锁   |
| --------------- | ------ | ------ |
| S 锁            | 相容   | 不相容 |
| X 锁            | 不相容 | 不相容 |

解析：
共享锁之间相容，因为多个事务可以同时读同一个数据对象；排他锁与任何锁都不相容，因为修改数据时必须保证独占访问。

### 3. 题目

简述数据库故障的主要分类及各类故障的成因与恢复方式。

答案与解析：

数据库故障主要包括事务故障、系统故障、介质故障和计算机病毒等。

1. 事务故障

   成因：
   某个事务在运行过程中因输入错误、运算溢出、违反完整性约束、死锁等原因不能继续执行。

   恢复方式：
   撤销该事务已经对数据库所做的修改，即执行 UNDO，使数据库恢复到事务开始前的状态。

2. 系统故障

   成因：
   操作系统故障、DBMS 故障、停电、硬件瞬时故障等，导致系统停止运行，但磁盘数据一般没有破坏。

   恢复方式：
   系统重新启动后，根据日志文件：
   - 撤销故障发生时尚未提交的事务；
   - 重做故障发生前已经提交但其结果可能尚未写入数据库的事务。

3. 介质故障

   成因：
   磁盘损坏、磁头碰撞、强磁干扰等导致数据库或日志文件部分或全部丢失。

   恢复方式：
   利用最近一次数据转储副本恢复数据库，然后利用日志文件重做转储以后已经提交的事务。

4. 计算机病毒

   成因：
   恶意程序破坏数据库文件或日志文件。

   恢复方式：
   先清除病毒，再利用备份副本和日志文件进行恢复。实际系统中还需要加强安全控制和备份管理。

## 四、关系代数及 SQL 语句（4 分/题 \* 5 题 = 20 分）

已知关系：

供应商表 Supplier（供应商编号 Sno，供应商名称 Sname，所在城市 City）

零件表 Part（零件编号 Pno，零件名称 Pname，零件规格 Spec）

供应表 Supply（供应商编号 Sno，零件编号 Pno，供应数量 Qty）

### 1. 题目

查询所有位于“西安”的供应商名称和供应商编号。

答案：

关系代数表达式：

$\pi_{Sname, Sno}(\sigma_{City='西安'}(Supplier))$

解析：
先用选择运算 $\sigma$ 选出 City 为“西安”的供应商，再用投影运算 $\pi$ 取出供应商名称和供应商编号。

### 2. 题目

查询供应了 P1 零件的供应商名称、所在城市及供应数量。

答案：

关系代数表达式：

$\pi_{Sname, City, Qty}(\sigma_{Pno='P1'}(Supplier \bowtie_{Supplier.Sno=Supply.Sno} Supply))$

解析：
先将 Supplier 与 Supply 按供应商编号连接，再选择 Pno 为 P1 的元组，最后投影出供应商名称、所在城市和供应数量。

### 3. 题目

查询供应商 S1 供应的零件总数（统计字段命名为 Total_Qty）。

答案：

SQL Server 语句：

```sql
SELECT ISNULL(SUM(Qty), 0) AS Total_Qty
FROM Supply
WHERE Sno = 'S1';
```

解析：
SUM(Qty) 用于统计供应数量总和。使用 ISNULL 是为了避免 S1 没有供应记录时返回 NULL。

### 4. 题目

查询供应数量大于 100 的 P2 零件的供应商名称、零件规格及供应数量。

答案：

SQL Server 语句：

```sql
SELECT
    s.Sname,
    p.Spec,
    sp.Qty
FROM Supplier AS s
JOIN Supply AS sp
    ON s.Sno = sp.Sno
JOIN Part AS p
    ON sp.Pno = p.Pno
WHERE sp.Pno = 'P2'
  AND sp.Qty > 100;
```

解析：
该查询需要同时使用供应商表、零件表和供应表。通过 Sno 连接供应商和供应关系，通过 Pno 连接零件和供应关系。

### 5. 题目

查询没有供应零件 P3 的供应商信息。

答案：

SQL Server 语句：

```sql
SELECT
    s.Sno,
    s.Sname,
    s.City
FROM Supplier AS s
WHERE NOT EXISTS (
    SELECT 1
    FROM Supply AS sp
    WHERE sp.Sno = s.Sno
      AND sp.Pno = 'P3'
);
```

解析：
NOT EXISTS 表示不存在对应记录。该语句查询在 Supply 表中不存在供应 P3 记录的供应商。

### 3. 题目

指出上述关系模式分别为第几范式，为什么。（7 分）

答案：

在以下基本函数依赖假设下：

1. CollegeID → CollegeName
2. MajorID → MajorName, CollegeID
3. StudentID → StudentName, MajorID
4. CourseID → CourseName, CollegeID
5. TeacherID → TeacherName, CollegeID
6. (StudentID, CourseID) → Grade
7. (TeacherID, CourseID) → Classroom, Hours

各关系模式范式如下：

1. College(CollegeID, CollegeName)

   范式：BCNF

   原因：
   其唯一非平凡函数依赖为 CollegeID → CollegeName，决定因素 CollegeID 是候选码。

2. Major(MajorID, MajorName, CollegeID)

   范式：BCNF

   原因：
   MajorID 是主键，MajorName 和 CollegeID 都完全函数依赖于 MajorID，不存在部分依赖和传递依赖。

3. Student(StudentID, StudentName, MajorID)

   范式：BCNF

   原因：
   StudentID 是主键，StudentName 和 MajorID 都完全函数依赖于 StudentID，且决定因素是候选码。

4. Course(CourseID, CourseName, CollegeID)

   范式：BCNF

   原因：
   CourseID 是主键，CourseName 和 CollegeID 都完全函数依赖于 CourseID，不存在非主属性对码的部分依赖和传递依赖。

5. Teacher(TeacherID, TeacherName, CollegeID)

   范式：BCNF

   原因：
   TeacherID 是主键，TeacherName 和 CollegeID 都完全函数依赖于 TeacherID，决定因素是候选码。

6. SC(StudentID, CourseID, Grade)

   范式：BCNF

   原因：
   主键为 (StudentID, CourseID)，Grade 表示某学生选修某课程的成绩，完全函数依赖于整个主键：

   $(StudentID, CourseID) \to Grade$

   Grade 不依赖于 StudentID 或 CourseID 的某一部分，因此不存在部分函数依赖，也不存在传递函数依赖。

7. Teaching(TeacherID, CourseID, Classroom, Hours)

   范式：BCNF

   原因：
   主键为 (TeacherID, CourseID)，Classroom 和 Hours 表示某教师讲授某课程时的教室和课时信息，完全函数依赖于整个主键：

   $(TeacherID, CourseID) \to Classroom, Hours$

   因此不存在部分依赖和传递依赖，且决定因素是候选码。

总结：
上述关系模式在属性原子性的基础上，非主属性均完全依赖于候选码，且不存在非主属性对候选码的传递依赖；进一步看，每个非平凡函数依赖的决定因素都是候选码，因此均可认为达到 BCNF。

## 六、SQL 编程（共 10 分）

### 题目

假设存在一个员工信息表 Employees（emp_id, emp_name, dept_id, salary），请编写一个 SQL 存储过程，要求如下：

存储过程名称：proc_get_employee_name;

输入参数：in_emp_id INT（员工 ID）；

输出参数：out_emp_name VARCHAR(50)（员工姓名）；

功能要求：

若输入的员工 ID 存在，将该员工的姓名赋值给输出参数；

若输入的员工 ID 不存在或为 NULL，输出参数赋值为“无此员工”。

答案：

SQL Server 存储过程：

```sql
IF OBJECT_ID(N'dbo.proc_get_employee_name', N'P') IS NOT NULL
BEGIN
DROP PROCEDURE dbo.proc_get_employee_name;
END;
GO

CREATE PROCEDURE dbo.proc_get_employee_name
@in_emp_id INT,
@out_emp_name VARCHAR(50) OUTPUT
AS
BEGIN
SET NOCOUNT ON;

    IF @in_emp_id IS NULL
       OR NOT EXISTS (
            SELECT 1
            FROM dbo.Employees
            WHERE emp_id = @in_emp_id
       )
    BEGIN
        SET @out_emp_name = '无此员工';
        RETURN;
    END;

    SELECT @out_emp_name = emp_name
    FROM dbo.Employees
    WHERE emp_id = @in_emp_id;

END;
GO
```

解析：

1. IF OBJECT_ID(...) IS NOT NULL 用于判断存储过程是否已经存在，若存在则先删除，避免重复创建时报错。

2. @in_emp_id 是输入参数，@out_emp_name 是输出参数。

3. 如果 @in_emp_id 为 NULL，或者 Employees 表中不存在该员工编号，则将输出参数赋值为“无此员工”。

4. 如果员工存在，则通过 SELECT @out_emp_name = emp_name 将员工姓名赋值给输出参数。

调用示例：

```sql
DECLARE @name VARCHAR(50);

EXEC dbo.proc_get_employee_name
@in_emp_id = 1,
@out_emp_name = @name OUTPUT;

SELECT @name AS emp_name;
```
