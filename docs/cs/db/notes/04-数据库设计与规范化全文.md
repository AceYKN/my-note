## 规范化设计通用做题流程

课堂流程：

```text
规范化设计题目：
书本上做题流程：

1. 形式化：问题描述（自然语言）。
2. 应用语义归纳：补充潜在语义/默认语义。
3. 形式化定义：R<U, F>。
   其中：
   U = { ... }
   F = { ... }
   码：
   非主属性集 = { ... }
4. 说明范式，说明不合理性。
5. 规范化推导（到 BCNF）。
6. 规范化结果：
   R1(...), 码：..., 外码：...
   R2(...), 码：..., 外码：...

经过以上规范化推导，问题已经设计到 BCNF，
达到函数依赖范畴的最高级范式。
```

考试项目设计模式：

```text
1. 形式化：问题描述（自然语言）。
2. 应用语义归纳：补充潜在语义/默认语义。
3. 绘制 E-R 图。
4. 转换关系模式。
5. 说明范式，说明不合理性。
6. 规范化推导（到 BCNF）。
7. 给出设计结果：
   R1(...), 码：..., 外码：...
   R2(...), 码：..., 外码：...
```

## 面向农业的电子商务系统规范化题

题面：

```text
五、数据库规范化：（共15分）

面向农业的电子商务系统部分需求概述：
注册客户能看见农户基本信息、生产场地基本信息、
生产的农产品及规格、价格、产能等信息，
能查询往年生产统计信息：
乡镇、镇长电话、农产品、总产能、平均价格。

要求：
1）用 E-R 图设计尽可能好的概念模型。（4分）
2）将 E-R 图转换成关系模型。（3分）
3）写出各个关系的函数依赖关系。（3分）
4）写出各个关系的主码和外码。（2分）
5）写出各个关系满足的范式级别。（2分）
6）将规范级别低的关系转成高级别范式的关系。（1分）
```

做题方法：

1. 找实体：客户、农户、生产场地、农产品、规格、乡镇、年度生产统计。
2. 找联系：
   - 客户查看农户/产品信息。
   - 农户拥有生产场地。
   - 农户生产农产品。
   - 乡镇统计农产品年度产能和平均价格。
3. 关系模式示例：

```text
Customer(CustomerID, CustomerName, ...)
Town(TownID, TownName, MayorPhone)
Farmer(FarmerID, FarmerName, TownID, ...)
ProductionSite(SiteID, FarmerID, SiteName, Address, ...)
Product(ProductID, ProductName, Spec)
FarmerProduct(FarmerID, ProductID, SiteID, Price, Capacity)
AnnualStat(TownID, ProductID, Year, TotalCapacity, AvgPrice)
```

4. 主码/外码示例：
   - `Town(TownID)` 主码。
   - `Farmer.TownID` 外码引用 `Town`。
   - `ProductionSite.FarmerID` 外码引用 `Farmer`。
   - `FarmerProduct(FarmerID, ProductID, SiteID)` 可作联合主码。
   - `AnnualStat(TownID, ProductID, Year)` 可作联合主码。

5. 范式说明：
   - 拆分后每个非主属性完全依赖本关系主码。
   - 乡镇电话等不放入统计关系中，避免 `TownID -> MayorPhone` 造成传递依赖。
   - 一般可说明达到 3NF/BCNF，具体取决于写出的函数依赖。

题面：

```text
五、数据库规范化：（共15分）

设某商品-销售数据库中的信息有：
员工号、员工名、工资、销售组名、销售组负责人、
商品号、商品名/商品价（原图局部作“商品价/单价”）、销售日期、销售量、
供应者号、供应者名、供应者地址。

假定：
1. 一个员工仅在一个销售组。
2. 一个销售组可销售多种商品。
3. 一种商品只能由一个组销售。
4. 一种商品每天有一个销售量。
5. 一个供应者可以供应多种商品。
6. 一种商品可以多渠道供货。

要求完成：
1）绘制 E-R 图设计概念模型。（4分）
2）将 E-R 图转换成关系模型。（2分）
3）写出各个关系的函数依赖关系。（2分）
4）写出各个关系的主码和外码。（5分）
5）写出关系满足的范式级别，并转成高级别范式。（2分）
```

参考实体与联系：

```text
实体：
Employee(员工)
SalesGroup(销售组)
Product(商品)
Supplier(供应者)
SaleRecord(销售记录)
Supply(供应联系)

联系：
Employee 多对一 SalesGroup
SalesGroup 一对多 Product
Product 一对多 SaleRecord
Supplier 多对多 Product，通过 Supply 表
```

## 高校教学管理信息系统设计题

题面：

```text
五、数据库设计（共20分）

某高校拟开发教学管理信息系统，用于管理学生、专业、课程、教师、学院
等核心业务对象，系统需满足以下业务需求：

学生与专业：
每个学生主修一个专业，一个专业可招收多名学生；
学生需选修多门课程，每门课程可被多名学生选修，
需记录学生每门课程的选修成绩。

专业与学院：
每个学院可设置多个专业，每个专业隶属于一个学院。

课程与学院：
每门课程由一个学院开设，一个学院可开设多门课程。

课程与教师：
一门课程可由多名教师讲授，一名教师可讲授多门课程，
需记录每门课程的授课教室与课时信息。

教师与学院：
每名教师隶属于一个学院，一个学院可拥有多名教师。

题目：
1. 根据上述业务需求，完成该系统的概念结构设计，绘制对应的 E-R 图。
   要求标注所有实体型、联系型；标注每个联系的类型（1:1、1:n、m:n）；
   标注联系的属性（如成绩、教室、课时）。（6分）
2. 将该 E-R 图转换为关系数据库的关系模式。
   要求标注每个关系模式的主键和外键。（7分）
3. 指出上述关系模式分别为第几范式，为什么。（7分）
```

文本化 E-R：

```text
学院 1 --- n 专业
学院 1 --- n 课程
学院 1 --- n 教师
专业 1 --- n 学生
学生 m --- n 课程，通过 选课 联系，联系属性：成绩
教师 m --- n 课程，通过 授课 联系，联系属性：教室、课时
```

关系模式：

```text
College(CollegeID, CollegeName)
Major(MajorID, MajorName, CollegeID)
Student(SID, SName, MajorID)
Course(CID, CName, CollegeID)
Teacher(TID, TName, CollegeID)
SC(SID, CID, Grade)
TC(TID, CID, Classroom, Hours)
```

主键与外键：

```text
College: PK(CollegeID)
Major: PK(MajorID), FK(CollegeID) -> College
Student: PK(SID), FK(MajorID) -> Major
Course: PK(CID), FK(CollegeID) -> College
Teacher: PK(TID), FK(CollegeID) -> College
SC: PK(SID, CID), FK(SID) -> Student, FK(CID) -> Course
TC: PK(TID, CID), FK(TID) -> Teacher, FK(CID) -> Course
```

范式说明：

- 每个关系属性原子，满足 1NF。
- 非主属性完全依赖主键，满足 2NF。
- 非主属性不传递依赖主键，满足 3NF。
- 若每个函数依赖左部都是候选码，可进一步说明满足 BCNF。

## 全球包裹递送公司设计题

题面：

```text
Relational Database Design

Design a database for a worldwide package delivery company
(e.g., EMS or SF Express).

The database must be able to keep track of customers who ship items
and customers who receive items; some customers may do both.

Each package must be identifiable and trackable, so the database
must be able to store the location of the package and its history
of locations. Locations include trucks, planes, airports, and warehouses.

1. Design an E-R diagram.
2. Design a set of relational schemas, and a list of constraints,
   including primary key and foreign key constraints.
3. Explain what normal form each relational schema satisfies, and why.
```

关系模式示例：

```text
Customer(CustomerID, Name, Phone, Address)
Package(PackageID, SenderID, ReceiverID, Weight, CreatedAt, CurrentLocationID)
Location(LocationID, LocationType, Description)
Truck(LocationID, PlateNo)
Plane(LocationID, FlightNo)
Airport(LocationID, AirportCode, AirportName)
Warehouse(LocationID, WarehouseName, Address)
PackageLocationHistory(PackageID, LocationID, ArriveTime, LeaveTime)
```

约束：

```text
Customer: PK(CustomerID)
Package: PK(PackageID),
         FK(SenderID) -> Customer(CustomerID),
         FK(ReceiverID) -> Customer(CustomerID),
         FK(CurrentLocationID) -> Location(LocationID)
Location: PK(LocationID)
Truck/Plane/Airport/Warehouse: PK/FK(LocationID) -> Location(LocationID)
PackageLocationHistory:
         PK(PackageID, LocationID, ArriveTime),
         FK(PackageID) -> Package(PackageID),
         FK(LocationID) -> Location(LocationID)
```

范式说明：

- 将不同地点类型拆成子表，避免在一个大地点表中出现大量空值。
- 历史位置使用独立表，避免包裹表中重复存储多个历史地点。
- 各表非主属性依赖主键，通常可说明达到 3NF/BCNF。

## 学生与宿舍关系讨论

课堂问题清单：

```text
讨论：学生和宿舍的关系
实践应用：
1. 查询所有学生的住宿情况。
2. 查询住宿学生的住宿情况。
3. 查询学生的住宿的宿舍情况。
4. 查询所有宿舍的学生住宿情况。
5. 查询未住宿的宿舍情况。
6. 查询所有宿舍的学生住宿情况。
7. 查询所有学生和宿舍的住宿情况。
8. 查询待分配宿舍的学生情况。
9. 查询可分配宿舍的情况（假设一个宿舍最多 5 人）。
10. 给出待分配学生的住宿的方案。
```

方法：

- 查询所有学生住宿情况：学生表左连接住宿/宿舍表。
- 查询所有宿舍学生情况：宿舍表左连接住宿表和学生表。
- 同时保留未住宿学生和空宿舍：全外连接，或用 `UNION` 合并左右外连接。
- 查询可分配宿舍：按宿舍分组统计人数，筛选人数小于 5。

示例模式：

```text
Student(Sno, Sname, ...)
Dorm(DormID, Building, RoomNo, Capacity)
Live(Sno, DormID, BedNo)
```

示例 SQL：

```sql
SELECT s.Sno, s.Sname, d.Building, d.RoomNo, l.BedNo
FROM Student AS s
LEFT JOIN Live AS l ON s.Sno = l.Sno
LEFT JOIN Dorm AS d ON l.DormID = d.DormID;

SELECT d.DormID, d.Building, d.RoomNo, COUNT(l.Sno) AS UsedBeds
FROM Dorm AS d
LEFT JOIN Live AS l ON d.DormID = l.DormID
GROUP BY d.DormID, d.Building, d.RoomNo, d.Capacity
HAVING COUNT(l.Sno) < d.Capacity;
```

文本化流程：

```text
应用程序 -> 外模式 -> 模式 -> 内模式 -> OS -> 数据库
应用程序工作区 与 DBMS/系统缓冲区之间进行数据交换。
```

要点：

- 外模式面向用户/应用。
- 模式是数据库全局逻辑结构。
- 内模式描述物理存储。
- DBMS 通过三级模式和两级映像实现数据独立性。
